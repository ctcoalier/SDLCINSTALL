Dev Guides

Practical how-tos for building with Claude Code — start here, then bring your questions to the next meetup.

Getting started with Claude Code

Go from zero to your first AI-assisted change.

  1. Install Claude Code for your environment — the terminal CLI, desktop app, web, or an IDE extension (VS Code / JetBrains).
  2. Open it inside a project directory so it has your code as context.
  3. Ask for something concrete: "explain how routing works in this app".
  4. Make a small change request and review the diff before accepting.
  5. Commit early and often so you can always roll back.

Writing effective prompts for coding

Get sharper results with a little structure.

  1. State the goal and the constraints ("don't change the public API").
  2. Point at the relevant files or symbols instead of describing them.
  3. Ask for a plan first on anything non-trivial, then approve it.
  4. Request tests alongside the change, not after.
  5. Iterate in small steps — review each diff rather than one giant one.

Using skills & slash commands

Package repeatable workflows Claude can invoke.

  1. Type / to see available slash commands and skills.
  2. Use a skill when you have a multi-step workflow you repeat often.
  3. Author your own: give it a clear name, description, and steps.
  4. Keep skill descriptions specific so they trigger at the right time.
  5. Share useful skills with the group — they install via plugins.

MCP servers in 5 minutes

Connect Claude to external tools and data.

  1. MCP (Model Context Protocol) is a standard way for Claude to use external tools — browsers, databases, APIs.
  2. Pick a server for what you need (e.g. a browser or a docs server).
  3. Register it so Claude Code can discover its tools.
  4. Ask Claude to use it: "use the browser to open this URL".
  5. Start with one server; add more as your workflows grow.

A safe review-and-ship workflow

Move fast without breaking your main branch.

  1. Work on a feature branch, never directly on main.
  2. Ask Claude to implement in small, reviewable steps.
  3. Run your tests (or have Claude write them) before committing.
  4. Read every diff — you are the reviewer of record.
  5. Open a PR and let a teammate (or a review tool) take a second look.
